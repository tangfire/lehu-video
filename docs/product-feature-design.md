# 校园 e站功能设计总览

这份文档从产品和功能角度解释校园 e站。它回答三个问题：学生能用它做什么，运营后台能管什么，系统为什么这样设计。

技术架构见 `docs/architecture.md`，接口见 `docs/api-map.md`，运营后台细节见 `docs/admin-operations.md`。

## 产品定位

校园 e站第一阶段是一个面向深圳职业技术大学深汕校区的轻量校园社区。核心目标不是做泛社交平台，而是先把校园信息流、问答、反馈、通知和运营治理跑通。

首发主线：

- 学生端：看校园帖子、发文字/图片帖、评论互动、收藏、反馈、接收通知、使用 e仔问答。
- 运营端：供给内容、处理审核/举报/反馈、管理用户和权限、维护 e仔人设和知识库、看安全和日志。
- 自动化：低风险内容无感通过，不确定内容飞书提醒；e仔/RAG 和运营 Agent 帮运营降低人工盯后台成本。

## 设计原则

| 原则 | 说明 |
| --- | --- |
| 低门槛发布 | 用户发帖后立即看到成功，自己的帖子可见，公共展示由异步审核控制 |
| 少打扰运营 | 普通数据进后台，举报、重要反馈、审核拿不准、预算预警才推飞书 |
| 低成本首发 | 不做视频，不让图片走服务器带宽，AI 规则先行降低模型调用 |
| 可治理社区 | 公开内容必须可审核、可下架、可举报、可通知用户 |
| AI 可复盘 | e仔回复、RAG 命中、Agent 判断、模型成本都有后台记录 |
| 渐进扩展 | 先跑深汕校区，后续再加校区标签、活动日历、更多运营玩法 |

## 用户角色

| 角色 | 主要目标 | 主要入口 |
| --- | --- | --- |
| 普通学生 | 浏览校园内容、发帖、评论、收藏、反馈、查看通知 | 微信小程序 |
| e仔官方账号 | 在评论区回答校园问题，承接知识库能力 | 后台配置 + 自动任务 |
| 运营人员 | 管内容、处理举报反馈、发官方内容、生成朋友圈素材 | 运营后台 |
| 管理员 | 管权限、看安全、改审核和 Agent 开关 | 运营后台 |
| 值班 Agent | 巡检、建议、辅助审核 | `campus-agent` + 飞书 |

## 学生端功能

### 社区首页

社区首页承接“今天学校里发生什么”和“大家正在讨论什么”。

核心能力：

- 按分类浏览帖子。
- 支持文字帖和图片帖。
- 支持热度/时间等排序。
- 公开列表只展示已通过内容。
- 支持点赞、收藏、评论数展示。

暂不做：

- 短视频流。
- IM 私聊、好友、群聊。
- 复杂推荐算法。

### 发帖

发帖只支持文字和图片。视频入口继续关闭。

发布体验采用“作者可见优先”：

1. 用户填写内容和图片。
2. 图片通过预签名 URL 直传 COS/MinIO。
3. 调用发帖接口后立即返回成功。
4. 作者可以在详情和“我的帖子”看到新帖。
5. 公共首页只展示 `visible` 内容。
6. 异步审核通过后进入公共流；需修改时给作者站内消息。

小程序应优先展示这些用户友好字段：

| 字段 | 用途 |
| --- | --- |
| `publish_state` | `visible/syncing/needs_attention/hidden` |
| `public_visible` | 是否已公开展示 |
| `client_status_label` | 给用户看的短状态，例如“已发布”“同步中”“需修改” |
| `client_status_detail` | 我的帖子页短说明 |

不要在学生端直接展示“风控”“审核拒绝”“AI 判断”等后台词。

### 帖子详情与互动

详情页是社区互动的主场：

- 查看图文内容。
- 评论和回复。
- 点赞、收藏。
- 举报帖子或评论。
- 在评论区 `@e仔` 触发 e仔回复。

待审核帖子：

- 作者本人可以打开。
- 非作者访问不可见。
- 不建议开放点赞/评论，避免未公开内容产生互动。

### 我的页面和通知

我的页面承接个人资产：

- 我的资料、头像、昵称。
- 我的帖子、我的评论、我的收藏。
- 系统通知、举报结果、内容处理结果。
- 反馈入口、关于我们、协议和规则入口。

通知设计：

- 举报提交后：给举报人发“举报已收到”。
- 举报处理后：给举报人发克制结果。
- 内容需修改/被下架：通知作者。
- 后台群发通知：用于官方公告。

### 反馈与联系我们

反馈入口用于收集问题和合作信息。

默认即时飞书的类型：

- `contact`：联系我们。
- `cooperation`：合作。
- `bug`：Bug。
- `content`：内容问题。

默认不即时飞书的类型：

- `suggestion`：普通建议，进入后台和日报。

这个设计是为了减少噪音：真正需要运营快速响应的内容进飞书，普通建议集中处理。

### e仔问答

e仔不是开放聊天机器人，而是校园社区里的官方问答助手。

触发方式：

- 用户在评论区 `@e仔` 或 `@深汕e仔`。
- 系统异步创建回复任务，不阻塞用户评论。
- 官方账号由环境变量固定；后台可关闭 AI 自动回复，关闭或模型不可用时改为通知官方账号人工接管。

回答策略：

- 校园事实类问题优先查知识库。
- 低置信度不强答，使用默认回复或不回复。
- 回复语气由后台人设配置控制。
- 所有回复任务、RAG 命中和质量标注都能在后台复盘。

## 运营后台功能

### 数据总览

给运营一个“今天怎么样”的入口：

- 今日访问、登录、发帖、评论、反馈。
- 待处理帖子、举报、反馈、AI 任务。
- 7 天趋势和关键状态。

该接口用 Redis 短 TTL 缓存，避免运营反复刷新打 MySQL。

### 内容工作台

内容工作台负责社区内容治理：

- 查帖、筛选、搜索。
- 审核待审帖子。
- 置顶、精选、下架、删除。
- 批量操作。
- 查看 AI/Agent 初审理由。

运营处理内容时，后台是真实状态；小程序只展示用户友好的发布状态。

### 反馈与举报

这个页面合并处理：

- 帖子举报。
- 评论举报。
- 用户反馈。
- 评论管理。

举报流程：

1. 用户举报。
2. 系统生成站内确认消息。
3. 系统写入 `campus_ops_alert`。
4. 飞书推送给运营。
5. 运营可在飞书点击下架/忽略，也可回后台处理。
6. 处理后通知举报人。

### 审核设置

审核有三种模式：

| 模式 | 行为 |
| --- | --- |
| 不审核 | 新帖直接公开，不创建 AI 审核任务 |
| 人工审核 | 作者可见，公共不可见，运营手动审核 |
| AI/Agent 初审 | 规则先行，低风险自动通过，中高风险进入 Agent 和飞书确认 |

图片内容审核第一阶段不接付费识别 API。低成本兜底策略是：

- 文本规则仍然先行；标题/正文命中高风险词或需复核词时，图片帖和文字帖一样进入待处理。
- 图片帖如果正文过短，会被规则判为需复核，不会因为只带图片就直接放行。
- 新用户、短文图片帖、被多次举报的图片内容优先人工查看；可信老用户的普通图片帖可以维持无感发布。
- 多次举报后可以先下架或隐藏，再由运营复核，避免违规图片长时间公开。
- 暂不建议在 2核4G 服务器上跑本地开源图片审核模型；它不花 API 钱，但会吃内存和 CPU，影响主链路。

Agent 开关：

- Agent 模型能力总开关。
- AI/Agent 发帖初审开关。
- 飞书运营通知总开关。
- 每日报告、高风险、举报、重要反馈子开关。
- AI 日/月预算和预警比例。

这些设置写入 `campus_ops_setting`，后台保存后生效，不需要重启服务。

### 运营 Copilot

运营 Copilot 是后台里的值班 Agent 页面，不是聊天页。

三类任务：

- 每日巡检：社区、审核、举报、反馈、e仔、RAG、安全。
- RAG 缺口：从错误标注和低置信日志里找资料缺口。
- 治理建议：根据待审、举报、反馈给处理优先级。

结果可以手动发送到飞书。高风险结果会自动提醒。

### e仔助手与知识库

e仔助手包含：

- 回复状态。
- 人设配置。
- 知识库文档。
- 知识库测试。
- RAG 评测。
- 失败任务。

RAG 质量闭环：

1. 用户真实问题生成查询日志。
2. 运营标注 `good/needs_fix/wrong/unsafe`。
3. Agent 自动沉淀停用评测草稿。
4. 运营确认后启用为评测用例。
5. 补资料、重建索引、批量评测。

### 朋友圈素材

后台可以生成朋友圈九图素材包：

- 自动取今日热帖，也支持手动选帖子。
- 生成 1080x1080 九宫格图片。
- 每张图带帖子内容和小程序码。
- 下载 ZIP 后由 e仔官方微信号手动发朋友圈。

不做自动发朋友圈，因为普通微信号/小程序没有开放后台自动发朋友圈能力。

### 安全中心

安全中心用于轻量风控和排障：

- 今日请求数、独立 IP、错误数。
- Top IP、Top Path。
- 最近访问日志。
- IP 封禁和解封。

真实 IP 限流依赖可信代理配置。普通容器日志不进 MySQL，排障主要看 Grafana/Loki。

### 权限管理

权限分两层：

- `admin`：管理权限、审核设置、系统配置。
- `operator`：日常运营、内容处理、通知和素材。

生产不要开启 `LEHU_CAMPUS_ADMIN_ALLOW_ALL=true`。

## 自动化与飞书闭环

飞书不是只发技术告警，也承接运营值班。

| 场景 | 是否飞书 | 处理方式 |
| --- | --- | --- |
| API/Redis/核心依赖不可用 | 是 | Grafana 告警；生产云 MySQL 通过 `api_ready` 间接覆盖 |
| RAG/Agent/alert-webhook 不可用 | 是，warning | Grafana 告警 |
| 用户举报 | 是 | 飞书下架/忽略或回后台 |
| 重要反馈 | 是 | 回后台处理 |
| 普通建议 | 默认否 | 后台和日报 |
| 审核不确定/高风险 | 是 | 飞书通过/拒绝或回后台 |
| AI 预算 70%/90% | 是 | 调整预算或关闭模型能力 |
| 每日巡检 | 是 | 查看摘要和跳转后台 |

如果飞书 webhook 未配置，本地环境会降级为记录日志，不影响主流程。

## 首发不做的功能

| 功能 | 暂不做原因 |
| --- | --- |
| 视频帖 | 成本、防刷、审核风险高 |
| IM 私聊/群聊 | 会显著增加治理和消息成本 |
| 图片内容审核 | 第一阶段先降低成本；靠举报、人工和文本审核兜底 |
| 自动发朋友圈 | 微信没有给普通微信号/小程序开放该能力 |
| 复杂推荐算法 | 数据量不足，先用时间、热度和运营精选 |
| 多校区强隔离 | 先跑深汕校区，后续再加校区标签和筛选 |

## 后续可扩展方向

优先级从高到低：

1. 多校区资料和用户标签：`campus_code/school_code`、主页标签、按校区筛选内容。
2. 校园日期/考试/活动日历：由运营维护，e仔可引用。
3. RAG 评测趋势：保存每次评测 run，用图表看质量变化。
4. 业务指标告警：上传失败率、5xx 率、AI 失败率。
5. 私有 COS：知识库文件从本地/内部链路迁到私有 COS。
6. 图片审核：等真实流量和成本压力明确后再接入。

## 首发验收清单

学生端：

- 微信真实登录成功。
- 发文字帖、图片帖成功。
- 发布后作者可见，公共流按审核状态展示。
- 评论、点赞、收藏、举报、反馈、通知可用。
- e仔可按人设回复，低置信度不乱答。

运营端：

- 后台登录和权限正常。
- 内容工作台、反馈举报、审核设置可用。
- Agent 开关、飞书通知、AI 预算可配置。
- e仔人设、知识库、RAG 评测可用。
- 朋友圈素材可生成和下载。

运维端：

- Grafana 日志搜索有数据。
- 健康面板包含 API、Redis、RAG、Agent、alert-webhook、Qdrant、Consul；生产云 MySQL 通过 `api_ready` 间接观察。
- 飞书能收到 Grafana 告警和运营通知。
- `campus_access_log` 清理任务生效。
- COS/CDN 上传下载链路正常。
